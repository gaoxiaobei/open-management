## 2. 编号规则

文档编号采用以下格式：

`OM-类别-子类-序号`

示例：

- `OM-REQ-US-001`
- `OM-REQ-FSD-001`
- `OM-DES-ARCH-001`
- `OM-DES-DB-001`

## 3. 文档状态定义

| 状态 | 说明 |
| --- | --- |
| 草案 | 内容正在编制，尚未进入正式评审 |
| 评审版 | 用于需求评审或设计评审的正式版本 |
| 基线版 | 已通过评审，作为实施和测试基线 |
| 归档版 | 历史版本，不再继续更新 |

## 4. 版本规则

- 重大范围调整或结构调整：升级主版本号
- 章节增强和新增内容：升级次版本号
- 错别字和格式修正：升级修订号

示例：

- `V1.0`
- `V1.1`
- `V2.0`

## 5. 变更流程

1. 需求或设计提出变更申请
2. 评估影响范围和关联文档
3. 更新对应文档内容
4. 在修订记录中登记版本和原因
5. 重新提交评审或纳入下一个基线

## 6. 一致性要求

- 术语、状态、模块命名在全套文档中保持一致
- 接口字段名和数据库字段名应遵循统一命名规范
- 用户故事、FSD、SRS、设计文档之间应保持可追踪关系

## 7. 评审建议

- 需求评审关注范围、角色、验收标准
- 设计评审关注边界、依赖、扩展性和安全性
- 实施评审关注计划、资源、风险和交付物
